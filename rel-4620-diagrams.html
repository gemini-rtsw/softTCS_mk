REL-4620 — Understanding the PWFS-2 Wrap Limit Bug

Updated April 2026 — incorporates monitoring data (Feb 2025), prior investigation (Sep 2025), and source code analysis. See rel-4620.md for the full write-up.

Part 1: What Is It Trying To Do?

The Real-World Goal

The telescope has guide probes (PWFS-1 and PWFS-2) mounted on rotary tables. As the telescope tracks a star across the sky, each probe slowly rotates to stay aligned. But the rotary table has physical stops — it can only spin so far before it hits a mechanical limit.

The TCS needs to answer one question: "How many minutes until this probe hits its limit?"

That countdown is displayed on the operator's screen (TSD). When it reaches zero, the system triggers an unwrap — it reverses the probe's rotation direction so it can keep tracking.

In plain terms: Imagine you're winding a garden hose reel. You can see the hose getting close to fully wound. You need to predict "how many more turns before it's maxed out?" and display a countdown so the operator knows when to reverse.

When does this code run? Only when the Cassegrain rotator is in fixed (AZEL_TOPO) mode and P2 is actively tracking — the configuration used for MAROON-X. Before MAROON-X was installed in 2021, this was an uncommon mode, which is why the bug went unnoticed for years.

What Information Does It Need?

The hidden inputs are the problem. The monitoring data proved that all 7 explicit EPICS inputs (limits, debug flag) had zero variation across sessions. But the function also reads global variables (probe position, sidereal time, pointing demands) that change every 50ms cycle. The algorithm is hypersensitive to these hidden changes.

Part 2: How the Current Code Solves It

The Algorithm in Plain Terms

The code uses a two-phase approach:

  1. Phase 1 — Solve for candidate times (trig): Convert the distance-to-limit into a parallactic angle, then use tcsLimPa() to find future hour angles where PA equals the limit PA. Returns 0, 1, or 2 candidate HAs per limit (up to 4 total for lo+hi).
  2. Phase 2 — Simulate forward to validate (loop): Step the hour angle forward in 6 coarse 90° jumps. At each step, project where the probe would be by accumulating parallactic angle changes onto the current encoder position. When you pass a candidate HA, check: "Is the projected probe position within 180° of the limit?" If yes, accept that time.

The simulation step exists because the trig in Phase 1 can return "ghost" solutions — hour angles where the PA value matches, but the probe would actually be at the opposite limit. The proximity check is supposed to reject those ghosts.

Part 3: Why It Breaks — The Two Bugs

Bug 1 (PRIMARY): Cumulative slaDrange() Error

This is the dominant cause. The PA accumulation across 6 coarse steps compounds rounding errors from slaDrange() boundary crossings, pushing p2RT far from reality.

Bug 2 (SECONDARY): The 180° Razor Edge

Even if Bug 1 were fixed, the proximity check has no hysteresis. When the probe is genuinely near the 180° crossover point, the check flips between accepting different solutions.

Part 4: The Fix — Direct Forward Integration

The Old Way vs. The New Way

The concept is simple calculus: The probe rotates because the parallactic angle changes as the Earth turns. The PA rate at any future moment is computable from trig (star position + hour angle + latitude). So: step forward in small time increments, compute the PA at each step, add up the rotation, and stop when it crosses a limit. That's it — area under the curve equals distance traveled.
Behavior change to communicate to ops: In geometries where the probe genuinely does not reach a limit within 24 h, the new algorithm reports invalid (-999, displayed as "−" on the TSD). The old code returned a large-but-finite number in those cases — it wasn't physically meaningful, but it looked like a countdown. A dash means "no limit in 24 h", not "system broken".

What the New Code Looks Like

What was removed: The tcsLimPa() calls for P1/P2, the candidate HA arrays, the P1/P2 portions of the 90° simulation loop, the cumulative slaDrange() accumulation (lines 12928-12929), and the 180° proximity checks. The azimuth and rotator wrap calculations are untouched.

How To Validate On Hardware

  1. Configure MAROON-X or equivalent fixed-CRCS P2-following setup (r_frame == AZEL_TOPO, p2Following == 1).
  2. Start tracking.
  3. Watch tcs:limits:times.VALJ on the TSD or via camonitor.
  4. Old bug signature: alternating values every update (e.g. 15/22/15/22…) or jumps of hundreds of minutes with constant inputs.
  5. Fixed signature: a smooth monotonically-decreasing countdown.

For verbose diagnostics, set INPH=1 on the times genSub record to enable the per-step integration trace in the IOC log.
