# REL-4620 — PWFS-2 Spurious Unwrap Fix

## 0. Current Status (2026-04-29)

**Two bugs identified, both fixed in this branch.**

1. **Algorithm fragility in `tcsLimitTimes`** — replaced the
   90°-step inverse-trig + 180° proximity hack with a clean
   forward-integration loop. Eliminates the chaos and 20-min
   granularity floor. *Fixed in `tcsPointing.c`.*
2. **1 Hz double-write race over the P2 rotary-table limits** in
   `tcsProbeLib.c` — two periodic writers were stomping on the same
   global, so the consumer (`tcsLimitTimes`) saw whichever value was
   current that millisecond. *Fixed by splitting the storage and
   combining deterministically on read.*

A diagnostic `printf` in the P2 block of `tcsLimitTimes` is left in
place to verify the next MAROON-X test produces a clean countdown.
It can be removed (or gated behind a debug flag) after that.

### What was wrong

- **`tcsLimitTimes` 90°-step algorithm:** 6 iterations of
  `slaDrange` accumulation could wrap and produce hundreds of
  degrees of cumulative error in the projected probe position. A
  separate 180° proximity hack tried to filter ghost solutions but
  used the corrupted projected position as the filter input. Tiny
  input jitter would flip the proximity check true/false, producing
  the alternating values and giant jumps users saw.
- **`tcsProbeLib.c` shared global:** the variable `pwfs2RTmin`
  (and `RTmax`) was written every 1 second by **two independent
  writers** holding the same mutex but writing **disagreeing
  values**:
  - `tcsProbeSetRTLimits`, called from `tcsSeqInit.st:1988`, wrote
    the hardware/calibration limits from `ag:p2:probeCalParm.{L,M}`.
  - `setRTCurrentLimits`, an EPICS subroutine wired in
    `tcsReadWfs.sch:21` at `SCAN:1 second`, wrote
    orientation-snapped limits (`-218/0`, `-180/180`, or
    `-218/218`) based on `ag:p2:setOrient.VALA`.

  Both are valid intents (mechanical bound + operator-selected
  orientation). The bug was that they shared the same memory slot,
  so the second writer overwrote the first and vice versa. The
  20 Hz `tcsLimitTimes` consumer read whichever was current at the
  moment of the call, giving rise to the bistable VALJ flip
  observed on-sky 2026-04-28 (142 ↔ 286 min) and in the 2025 GEA
  archive (480 ↔ 579 min).

### What was fixed

**`tcsPointing.c` — algorithm rewrite (`ab94016`, refined in
later commits):**

```c
#define P12_HA_STEP  (0.25 * D2R)                /* 1 sidereal min */
#define P12_N_STEPS  ((int)(PI2 / P12_HA_STEP))  /* 1440 steps = 24 h */

p2paLast = pa0;
p2rot    = 0.0;
for (i = 0; i < P12_N_STEPS; i++) {
    p2ha    = curha + (i + 1) * P12_HA_STEP;
    p2panew = slaPa(p2ha, dec, tlat);
    p2rot  += slaDrange(p2panew - p2paLast) * R2D;   /* tiny step → safe */
    p2paLast = p2panew;
    p2pos    = p2RTPosn + p2rot;
    if (p2pos <= p2RTLoLim) { p2ttl = (i+1)*P12_HA_STEP; p2valid=1; break; }
    if (p2pos >= p2RTHiLim) { p2ttl = (i+1)*P12_HA_STEP; p2valid=1; break; }
}
```

No `tcsLimPa` ghost solutions, no proximity hack, no cumulative
wrap error. 1440 steps × ~1 µs ≈ 1.4 ms per call (≈3 % of the 50 ms
update budget).

**`tcsProbeLib.c` — race fix:**

Split the racing globals into two slots and combine them on read.
External API (`tcsProbeSetRTLimits`, `setRTCurrentLimits`,
`tcsProbeGetRTLimits`) is unchanged so no SNL-side or EPICS-side
changes are required.

```c
/* Hardware / calibration limits — set by tcsProbeSetRTLimits */
static double pwfs1HwRTmin = -200.0,  pwfs1HwRTmax =  200.0 ;
static double pwfs2HwRTmin = -200.0,  pwfs2HwRTmax =  200.0 ;

/* Orientation override limits — set by setRTCurrentLimits.
 * Initialised wide so the hardware slot is unconstrained at startup. */
static double pwfs2OrRTmin = -360.0,  pwfs2OrRTmax =  360.0 ;
```

Reader combines (stricter wins):

```c
*p2RTMin = (pwfs2HwRTmin > pwfs2OrRTmin) ? pwfs2HwRTmin : pwfs2OrRTmin ;
*p2RTMax = (pwfs2HwRTmax < pwfs2OrRTmax) ? pwfs2HwRTmax : pwfs2OrRTmax ;
```

Behavior with each orientation is identical to before:

| `setOrient.VALA` | Or slot | Hw slot | Combined (effective) |
|---|---|---|---|
| 0 | (-218, 0) | (-218, +218) | (-218, 0) |
| 1 | (-180, +180) | (-218, +218) | (-180, +180) |
| default (≥2) | (-218, +218) | (-218, +218) | (-218, +218) |

Both intents are preserved. The mutex-protected read sees a
single deterministic combined value. No race possible.

**`tcsPointing.c` — diagnostic trace (temporary, to confirm fix):**

Two `printf` lines at the entry/exit of the P2 fixed-CRCS block
log every input the function reads and the output it produces.
Fires only in the bug-relevant path (`r_frame == AZEL_TOPO &&
p2Following`), so log volume is bounded (~20 lines/sec, ~30 KB/min).
After the next MAROON-X test confirms the fix produces a clean
countdown, the trace can be removed or gated behind the existing
`debug` flag.

### How this was validated

- **Algorithm:** 17/17 unit tests pass. Bit-exact stability under
  1000 identical-input calls. Smoke test of the new `printf` format
  confirms parseable output and correct gating.
- **Race fix:** A C-level test (`P2UnwrapFiles/tests/race_fix_test.c`)
  ran two 1 Hz writers at the production cadence with disagreeing
  values, and a 20 Hz reader. Old code: 2 distinct values, 59
  transitions in 30 s (flicker). New code: stable read, transitions
  only at startup as the orientation slot is first written. Fix is
  effective and deterministic.

### What still depends on the next on-sky test

Confirming that, with both fixes in place, an actual MAROON-X
observation produces a smooth `timeToP2lim` countdown. The
diagnostic trace is in place to prove it, and to catch any *other*
alternating input we haven't identified, in case the limit race
turns out not to have been the only contributor.

## 1. The Problem in One Paragraph

The TSD shows a countdown to when the PWFS-2 guide probe will hit its
rotary-table limit. In **fixed-CRCS mode with P2 following** (the
MAROON-X configuration), that countdown is unreliable: it either
oscillates between two values on every 50 ms update, or jumps by
hundreds of minutes with no real change in the inputs. When the
displayed countdown hits zero, the system triggers a spurious
"unwrap" — a disruptive motion that interrupts observing. Eight
on-sky incidents since 2021 trace back to this bug.

---

## 2. What The Countdown Is Computing

Function: **`tcsLimitTimes()`** in `tcs-mk-iocApp/src/tcs/tcsPointing.c`.

In fixed-CRCS mode the Cass rotator is held still, so the sky rotates
over the telescope as it tracks. To keep a guide star fixed on the
P2 probe, P2's rotary table must rotate at the **rate of change of
the parallactic angle (PA)**. Eventually the table reaches its
hardware limit. The countdown answers:

> Given the current probe position, how many minutes of sidereal time
> until the accumulated PA rotation drives it to `p2RTHiLim` or
> `p2RTLoLim`?

Result is written to genSub field `VALJ` in minutes. `-999` means
"no limit reached within the search horizon".

The probe-to-PA relationship is **1:1**: a guide star locked at any
fixed focal-plane (x, y) keeps the same radius from the optical axis
as the field rotates, so the AG inverse kinematics gives
`dα/dPA = +1.000` exactly across the entire focal plane. (See
`scripts/derive_scale_factor.py` in the P2UnwrapFiles/ tree for the
numerical proof; the analytic argument is that `α = atan2(-x,y) + pas(r)`
and `r` is invariant under field rotation.)

---

## 3. What Was Broken (Old Algorithm)

The old algorithm mixed three techniques, each of which introduced error:

1. **Inverse trig (`tcsLimPa`)** — solved for hour-angle candidates
   where the probe would reach a limit. Could return 0, 1, or 2
   "ghost" solutions per limit.
2. **Coarse forward simulation** — stepped HA in **90° chunks** (six
   steps, six sidereal hours) and accumulated PA changes step-by-step:
   ```c
   panew = slaPa(ha, dec, tlat);
   p2RT += slaDrange(panew - pa) * R2D;     // ← primary bug
   pa = panew;
   ```
   `slaDrange` wraps to ±180°. Over a 90° HA step, the PA change
   often exceeds that, and the wrap produces a large cumulative
   error in `p2RT`. After six steps, `p2RT` could be hundreds of
   degrees away from reality.
3. **180° proximity sanity check** — to filter out the ghost
   solutions, it only accepted a candidate if the (corrupted) `p2RT`
   was within 180° of the limit:
   ```c
   if (fabs(p2RT - p2RTLoLim) <= 180.0) {   // ← secondary bug
       /* accept this candidate */
   }
   ```
   Tiny input jitter flipped this check true/false, producing the
   alternating values and giant jumps users saw on the TSD.

**Root cause in one sentence:** a corrupted projected position
(`p2RT`) was used as a filter on ghost solutions, so the answer
depended on two layers of numerical garbage.

---

## 4. The Fix (New Algorithm)

Replaced the whole thing with straightforward **forward integration**:

```c
#define P12_HA_STEP  (0.25 * D2R)                // 0.25° = 1 sidereal minute
#define P12_N_STEPS  ((int)(PI2 / P12_HA_STEP))  // 1440 steps = 24 h

p2paLast = pa0;      // start: current parallactic angle
p2rot    = 0.0;      // start: zero accumulated rotation
for (i = 0; i < P12_N_STEPS; i++) {
    p2ha    = curha + (i + 1) * P12_HA_STEP;
    p2panew = slaPa(p2ha, dec, tlat);
    p2rot  += slaDrange(p2panew - p2paLast) * R2D;   // stays tiny — safe
    p2paLast = p2panew;
    p2pos   = p2RTPosn + p2rot;

    if (p2pos <= p2RTLoLim) { p2ttl = (i+1)*P12_HA_STEP; p2valid = 1; break; }
    if (p2pos >= p2RTHiLim) { p2ttl = (i+1)*P12_HA_STEP; p2valid = 1; break; }
}
```

### Why each piece matters

| Piece | Why |
|---|---|
| **0.25° step size** | 1 sidereal minute of HA. Small enough that the PA change per step is a fraction of a degree to a few degrees anywhere outside the zenith, so `slaDrange` never wraps. Also sets the countdown's 1-minute granularity. |
| **24 h range** | Full sky. If the probe doesn't hit a limit in 24 h, report invalid (`-999`) instead of guessing. |
| **Direct `p2pos <= lo` / `>= hi`** | No inverse trig, no ghost solutions, no proximity check. |
| **1:1 PA accumulation** | The AG inverse kinematics gives `dα/dPA = 1.000` everywhere in the focal plane (see Section 2). Accumulating raw `dPA` into `p2rot` is correct. |
| **1440 iterations of one `slaPa`** | ~1.4 ms per call, ~3% of the 50 ms update budget. |
| **Single break on first crossing** | Returns the *nearest* limit, which is what the operator needs. |

### About the 0.25° step size

**The step size is a choice, not a constraint.** 0.25° (exactly 1 sidereal
minute of HA) was picked to balance three things:

1. **Safety from `slaDrange` wrap.** `slaPa` is continuous and its rate
   is bounded outside the zenith singularity. A 0.25° HA step produces
   a PA change of only a fraction of a degree to at most a few degrees
   — far from the ±180° boundary that tripped up the old algorithm.

2. **Display quantization.** 0.25° of HA = **1 sidereal minute** of real
   time. That's the finest granularity the countdown can report, which
   matches the TSD's 1-minute display precision.

3. **CPU cost.** 24 h ÷ 0.25° = **1440 iterations**, each one `slaPa` call
   (a handful of `sin`/`cos`/`atan2`). Total ~1.4 ms per invocation,
   ~3% of the 50 ms update budget. Plenty of headroom.

The step size is tunable in a single `#define`. Don't change it unless
you have a specific reason — 0.25° is deep in the safe zone and gives
operator-usable 1-minute granularity. Going much larger (≥ 5°) starts
producing visible granularity floors; going ≥ 45° approaches the old
code's failure mode where `slaDrange` can wrap.

## 6. Behavior Change To Communicate

In geometries where the probe **genuinely does not reach a limit
within 24 h**, the new algorithm reports invalid (`-999`, displayed
as "−" on the TSD). The old code could return a large-but-finite
number in those cases — it wasn't physically meaningful, but it
looked like a countdown. Ops should know that a dash means "no
limit in 24 h", not "system broken".

## 7. How To Test On Hardware

The failure signature is easy to spot in a short test window:

1. Configure MAROON-X or equivalent fixed-CRCS P2-following setup
   (`r_frame == AZEL_TOPO`, `p2Following == 1`).
2. Start tracking.
3. Watch `tcs:limits:times.VALJ` on the TSD or via `camonitor`.
4. **Old bug signature:** alternating values every update (e.g.
   15/22/15/22...) or jumps of hundreds of minutes with constant
   inputs.
5. **Fixed signature:** a smooth monotonically-decreasing countdown.

For verbose diagnostics, set `INPH=1` on the `times` genSub record
to enable the `printf` output shown in the code. You'll see per-step
integration trace.

## 9.5. How the Race Got There — Code Archaeology

The race condition in `tcsProbeLib.c` was introduced in 2013 and lay
dormant for ~8 years before MAROON-X observing made it visible.
Source: the embedded RCS revision log at the top of the file.

### Original code (2008-05-01, Revision 1.12, by `cjm`)

> *"Routines to set and get rotary table limits, posns etc."*

The first version of the rotary-table limit machinery had:
- Globals `pwfs2RTmin`, `pwfs2RTmax` (and P1 equivalents).
- One writer: `tcsProbeSetRTLimits`, called from `tcsSeqInit.st`.
- One reader: `tcsProbeGetRTLimits`, called from `tcsLimitTimes`.

A single mutex protected the globals. No race possible. This
worked correctly for **five years**.

### The bug got planted: 2013-05-05 (Revision 1.24, REL-466, by `gemvx`)

> *"Modified Files: All related to REL-466.*
> *capfast/tcs/tcsReadAg.sch: Added gensub to set the PWFS2 RT limits*
> *according to the RT prefered orientation.*
> *capfast/tcs/tcsReadWfs.sch: Added p2RTorient record.*
> *src/tcs/tcsProbeLib.c: Added setRTCurrentLimits routine."*

REL-466 added a feature: let operators tighten the P2 rotary-table
limits depending on the *preferred orientation* of the probe (CW,
CCW, both) for cable-management reasons. Reasonable feature.

**The implementation introduced a second writer that wrote into the
same global slot as the first writer.** `setRTCurrentLimits` was
wired in EPICS at SCAN:1 second; `tcsProbeSetRTLimits` was already
being called from the SNL `monitoring` state every 1 second. Both
held the same mutex (so no torn reads), but they wrote *different
values* with no coordination. Whichever ran most recently won, until
the other ran again a fraction of a second later.

The `tcsLimitTimes` consumer reads the global at 20 Hz. From May
2013 onward, in fixed-CRCS + P2-following mode, it would read
whichever limit was current that millisecond — producing a bistable
flip whose magnitude depended on the difference between
`probeCalParm.M` and the orientation-snapped value (typically 38°
when orientation = 1, giving a ~152-min flip in TTL).

### Why the bug stayed hidden for ~8 years

1. **Fixed-CRCS + P2-following was a rare configuration.** This is
   the only path in `tcsLimitTimes` that consumes the limit globals.
   Most observing modes don't enter that branch and the racing
   global goes unread.

2. **MAROON-X arrived in 2021.** That instrument operates in
   fixed-CRCS mode by design. As soon as it started observing
   regularly, the race-induced flip began appearing in operator
   reports.

3. **Master's `tcsLimitTimes` algorithm produced its own chaos.**
   The 6×90° step + cumulative `slaDrange` wrap + ghost-solution
   filtering produced large random jumps even with stable inputs.
   That noise masked the cleaner two-value race signature, and
   investigators (correctly) focused on the algorithm.

### Why REL-4620 fixed both

Cleaning up the algorithm (commit `ab94016`) made the race
*visible*. With stable inputs, the new forward-integration loop
produces a stable output. So when the on-sky 2026-04-28 test still
showed bistable flipping (142 ↔ 286), it could no longer be
attributed to algorithm chaos — the inputs themselves had to be
alternating. That observation is what led to the race being
identified and fixed (today's `tcsProbeLib.c` change).

## 10. Walk-Through: Old Code vs. New Code

This section walks through both the old algorithm and the new one step by
step, with the actual code snippets. Intended for someone reading the code
for the first time.

### Shared setup (same in both versions)

Both versions read the current telescope state before doing any limit
calculation:

```c
/* Read probe state (current rotary-table position and hw limits). */
tcsProbeGetFollowFlag(0, &p1Following);
tcsProbeGetFollowFlag(1, &p2Following);
tcsProbeGetRTPosns(&p1RTPosn, &p2RTPosn);
tcsProbeGetRTLimits(&p1RTLoLim, &p1RTHiLim, &p2RTLoLim, &p2RTHiLim);

/* Current parallactic angle at the telescope's current hour angle. */
pa0 = slaPa(curha, dec, tlat);
```

Both versions then gate on the same condition:

```c
if (r_frame == AZEL_TOPO && p2Following) {
    /* ...P2 time-to-limit calculation... */
}
```

That's the MAROON-X / fixed-CRCS configuration. Outside this branch, P2
isn't rotating to track field, so the countdown isn't needed.

---

### The OLD algorithm — what it did, step by step

**Step 1 — Convert distance-to-limit into a parallactic angle.**
The old code thought "the probe reaches its limit when the PA equals some
target value." It computed that target PA by adding the distance-to-limit
to the current PA:

```c
dRTHiLim = p2RTHiLim - p2RTPosn;  /* degrees to high limit */
dRTLoLim = p2RTLoLim - p2RTPosn;  /* degrees to low  limit */

pa = pa0 + dRTLoLim * D2R;        /* PA at which lo limit is reached  */
p2locount = tcsLimPa(cdec, sdec, pa, clat, slat, p2halo);

pa = pa0 + dRTHiLim * D2R;        /* PA at which hi limit is reached  */
p2hicount = tcsLimPa(cdec, sdec, pa, clat, slat, p2hahi);
```

**Step 2 — Solve an inverse trig equation for hour-angle candidates.**
`tcsLimPa()` answers: *"at what future hour angles does the parallactic
angle equal this value?"* The trig has 0, 1, or 2 solutions per limit —
so up to **four candidate HAs** in total (lo × {sol1, sol2} + hi ×
{sol1, sol2}). These candidates are stored in `p2halo[]` and `p2hahi[]`.

The problem: some of these are "ghost" solutions — the math says the PA
would match there, but physically the probe would be at the opposite
limit, not the one we asked about. Something has to filter them out.

**Step 3 — Forward-simulate in coarse 90° HA steps, validating candidates.**
To distinguish real solutions from ghosts, the old code walked the hour
angle forward in 90° chunks (six steps = six sidereal hours) and
tracked where the probe *would* be at each step:

```c
p2RT = p2RTPosn;     /* start at the current probe position        */
pa   = pa0;          /* start at the current parallactic angle     */
ha   = floor(curha/D90) * D90;

for (i = 0; i < 6; i++) {           /* 6 steps × 90° = 6 hr ahead */
    ha += D90;                       /* advance HA by 90°          */
    panew = slaPa(ha, dec, tlat);    /* new PA at this step         */

    /* *** PRIMARY BUG *** */
    p2RT += slaDrange(panew - pa) * R2D;   /* accumulate probe rotation */
    pa    = panew;
```

`slaDrange` wraps its argument into (-180°, +180°). Over a 90° HA step
the PA change can be much larger than 180° (especially near the zenith),
so `slaDrange` gets called with an argument near its wrap boundary. At
floating-point precision, tiny input noise flips which side of the
boundary it lands on — and the error is **cumulative** across all six
steps. After the loop, `p2RT` can be hundreds of degrees away from
physical reality.

**Step 4 — At each candidate HA, sanity-check with a 180° proximity test.**
When the simulation passed one of the candidate HAs from Step 2, the
code checked whether the corrupted `p2RT` was within 180° of the limit:

```c
    for (j = 0; j < p2locount; j++) {
        if (!p2lodone[j] && p2halo[j] <= ha) {
            p2lodone[j] = 1;

            /* *** SECONDARY BUG *** */
            if (fabs(p2RT - p2RTLoLim) <= 180.0) {
                if (!p2valid || (p2halo[j] - curha) < p2ttl) {
                    p2ttl   = p2halo[j] - curha;  /* accept candidate */
                    p2valid = 1;
                }
            }
        }
    }
    /* ...same block for p2hicount / p2RTHiLim... */
}
```

Because `p2RT` was corrupted in Step 3, this "sanity check" was the
opposite of sane: it randomly accepted or rejected solutions based on
numerical garbage. Tiny input jitter flipped the decision between calls,
which is why the TSD alternated between two values or jumped by hundreds
of minutes.

**Step 5 — Pick the shortest accepted candidate, output to VALJ.**

```c
if (p2valid) {
    *(double *)pgsub->valj = p2ttl * ST2MIN;   /* to TSD */
}
```

No filtering, no hysteresis, no sanity bounds — whatever `p2ttl` was,
it flowed straight to the operator's display.

### The NEW algorithm — what it does, step by step

The new code deletes all four layers (inverse trig, ghost solutions,
coarse simulation, proximity check) and replaces them with one tight
integration loop.

**Step 1 — Set up small steps and a 24 h horizon.**

```c
#define P12_HA_STEP  (0.25 * D2R)               /* 0.25° = 1 sidereal min */
#define P12_N_STEPS  ((int)(PI2 / P12_HA_STEP)) /* 1440 steps = 24 h       */
```

Where the old code took 6 big 90° steps, the new code takes 1440 small
0.25° steps.

**Step 2 — Initialize the integration from the current state.**

```c
double p2paLast = pa0;     /* current parallactic angle (rad)   */
double p2rot    = 0.0;     /* total probe rotation so far (deg) */
```

**Step 3 — Step forward and integrate the probe rotation.**

```c
for (i = 0; i < P12_N_STEPS; i++) {
    double p2ha    = curha + (i + 1) * P12_HA_STEP;   /* next HA    */
    double p2panew = slaPa(p2ha, dec, tlat);          /* PA at p2ha */

    /* Add this step's PA change to the running total.
     * Because p2ha advances by only 0.25°, the change is small —
     * slaDrange's ±180° wrap is never crossed, so no cumulative
     * error.  dα/dPA = 1 from the AG inverse kinematics, so we
     * accumulate the raw PA change directly. */
    p2rot   += slaDrange(p2panew - p2paLast) * R2D;
    p2paLast = p2panew;

    /* Projected probe position at this step. */
    double p2pos = p2RTPosn + p2rot;
```

This is a simple Riemann sum: we're adding up `d(PA)` over small `d(HA)`
to get the total rotation.

**Step 4 — Check for a limit crossing directly, break on first hit.**

```c
    if (p2pos <= p2RTLoLim) {              /* hit the low limit */
        p2ttl   = (i + 1) * P12_HA_STEP;   /* elapsed HA        */
        p2valid = 1;
        break;
    }
    if (p2pos >= p2RTHiLim) {              /* hit the high limit */
        p2ttl   = (i + 1) * P12_HA_STEP;
        p2valid = 1;
        break;
    }
}
```

If the projected position crosses a limit, the elapsed HA *is* the
time-to-limit.

**Step 5 — If the loop finishes without a hit, report invalid.**
If we walked 24 h of future HA without the probe ever reaching a limit,
`p2valid` stays `0` and `VALJ` is set to `-999.0`. On the TSD this shows
as "−" — meaning "the probe will not reach a limit within the next 24 h".

**Step 6 — Same output contract as the old code.**

```c
if (p2valid) {
    *(double *)pgsub->valj = p2ttl * ST2MIN;  /* minutes, to TSD */
}
```

`ST2MIN` converts sidereal radians to minutes. The output field, the
unit, the validity sentinel, and the mask bit are all unchanged — that's
what makes the fix a true drop-in.

### Side-by-side summary

| Aspect | OLD | NEW |
|---|---|---|
| Approach | inverse trig + simulation + filter | forward integration |
| HA step size | 90° (6 steps, 6 h horizon) | 0.25° (1440 steps, 24 h horizon) |
| Per-step PA change | often > 180° → slaDrange wraps | always ~few° → safe |
| Candidate solutions | 0–4 per call, ghosts possible | 1, deterministic |
| Sanity check | 180° proximity on corrupted value | not needed |
| Failure mode | chaotic VALJ, spurious unwraps | none observed |
| CPU cost | 6 slaPa + 4 tcsLimPa + bookkeeping | 1440 slaPa (~1.4 ms) |
| Lines of logic | ~120 | ~30 |
