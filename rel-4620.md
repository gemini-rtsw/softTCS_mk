# REL-4620 — PWFS-2 Spurious Unwrap / Chaotic Time-to-Limit Calculation

## What's Happening (Plain English)

The telescope has a guide probe called PWFS-2 (P2) that sits on a rotary table.
As the telescope tracks a target across the sky, P2 slowly rotates toward its
mechanical limit — like a clock hand approaching 12. The TCS (Telescope Control
System) predicts *when* P2 will hit that limit and displays a countdown on the TSD
(Telescope Status Display).

**The bug:** The countdown displayed on the TSD is wildly wrong. Depending on
conditions, it manifests in two ways:

- **Oscillation mode:** The value alternates between two different times on every
  update — for example 15 minutes and 22 minutes, back and forth like a strobe.
  Eventually the shorter time reaches zero and triggers a spurious unwrap.
- **Chaos mode:** The value jumps by hundreds of minutes between successive
  50ms updates — e.g. from 416 to 579 minutes in a single cycle — even though
  every input parameter is constant. Three monitoring sessions on 2025-02-26
  showed averages of 4 min, 26 min, and 513 min with the same inputs.

This has caused at least 8 documented incidents since 2021:
- GNFR-73941, GNFR-73994, GNFR-74131, GNFR-74132 (2024 on-sky faults)
- GNFR-72825, GNFR-72665 (2022 reports, first to correlate with target stream errors)
- REL-4008 (2021 original implementation issue)
- GSFR-10114, GSFR-8107, GSFR-22068, GSFR-23948 (earlier P2 unwrap anomalies)

The problem is specific to **fixed CRCS mode with P2 tracking** — the configuration
used for MAROON-X observations (`r_frame == AZEL_TOPO && p2Following`).

## How the Countdown is Calculated

The function `tcsLimitTimes()` in `tcsPointing.c` (line 12602) does this:

1. **Read the current state** — P2's current rotary table position (`p2RTPosn`)
   from a live encoder, the current hour angle, declination, probe limits, etc.

2. **Compute the current parallactic angle** `pa0 = slaPa(curha, dec, tlat)`.
   The parallactic angle (PA) tells you how the sky is rotating relative to the
   telescope — it determines how fast the probe's rotary table is turning.

3. **Ask: "At what hour angle will P2 reach its limit?"** — It converts the
   distance-to-limit into a PA offset, then calls `tcsLimPa()` (in `tcsLimits.c`
   line 365) which solves a trig equation. This can return **0, 1, or 2 answers**
   — two different future hour angles where the math says the limit could be
   reached. It does this for both the low limit and the high limit, so there can
   be up to **four candidate answers**.

4. **Simulate forward in time** — Step the hour angle forward in 90° increments
   (6 steps covering 6 sidereal hours). At each step, project where P2 would be
   by accumulating parallactic angle changes:
   ```c
   panew = slaPa(ha, dec, tlat);
   p2RT += slaDrange(panew - pa) * R2D;   /* ← THE PRIMARY BUG */
   pa = panew;
   ```

5. **Check proximity** — When the simulation passes one of the candidate hour
   angles, check if the projected P2 position is actually close to the limit:
   ```c
   if ( fabs( p2RT - p2RTLoLim ) <= 180.0 ) {   /* ← SECONDARY BUG */
       /* Accept this time as the countdown value */
   }
   ```

6. **Pick the shortest valid time** and write it to `VALJ`, which flows through
   to the TSD display with zero filtering.

## Root Cause: Two Interacting Bugs

### Bug 1 (PRIMARY): Cumulative `slaDrange()` Error — Lines 12928-12929

**What it is:**
The forward simulation accumulates PA changes step-by-step:

```c
/* Lines 12928-12929 — THE PRIMARY BUG */
p1RT += slaDrange(panew - pa) * R2D;
p2RT += slaDrange(panew - pa) * R2D;
```

The function `slaDrange()` normalizes an angle into the range ±π (±180°). When
the PA difference `(panew - pa)` crosses the ±π boundary between steps — which
happens when the parallactic angle wraps around — `slaDrange()` maps a small
physical change into a large opposite-sign value. For example:

```
Real PA change: 179° → 181° (a +2° change)
(panew - pa) = 181° - 179° = +2°     → slaDrange returns +2°    ✓ OK

Real PA change: 179° → -179° (a +2° change that crosses the wrap)
(panew - pa) = -179° - 179° = -358°  → slaDrange returns +2°    ✓ OK in theory

But in practice, floating-point rounding near the boundary can make
slaDrange return -358° + 360° = +2°, or it might get -358° and return
+2°. The problem is CUMULATIVE: each step's rounding error adds to
p2RT permanently, and over 6 steps the errors compound.
```

Because the simulation uses **coarse 90° HA steps**, each step can span a
huge PA change, making boundary crossings common and the accumulated error
large.

**Evidence from monitoring data (2025-02-26):**
- All 7 EPICS input parameters were **perfectly constant** across all three
  monitoring sessions (zero variation)
- VALJ (P2 limit time) varied by 579 minutes total
- Other outputs (VALB elevation limit, VALC airmass limit) were stable
- Maximum single jump: **135.3 minutes in 0.1 seconds** with constant inputs
- Three distinct chaos levels observed:
  - Session 1: avg 4 min, range 9.7 min (MEDIUM chaos)
  - Session 2: avg 26 min, range 22.5 min (HIGH chaos)
  - Session 3: avg 513 min, range 162.8 min (EXTREME chaos)

**Why constant inputs produce different outputs:**
The function reads global variables (`St`, `M_tar_p[]`, `AzDef`, `RmaDef`,
`p2RTPosn`) that change between calls even though the EPICS *limit* inputs
are constant. Sidereal time advances, target stream errors create micro-
corrections in pointing demands, and the raw encoder position jitters. These
tiny changes in the hidden global state push the cumulative PA calculation
across different `slaDrange()` boundaries on each call, producing wildly
different accumulated `p2RT` values.

**The code path:**
```
Line 12731: tcsProbeGetRTPosns(&p1RTPosn, &p2RTPosn)   ← raw encoder, no filter
Line 12838: p2RT = p2RTPosn                              ← seed with noisy value
Line 12848: for (i = 0; i < 6; i++) {                    ← 6 coarse 90° steps
Line 12927:     panew = slaPa(ha, dec, tlat)
Line 12929:     p2RT += slaDrange(panew - pa) * R2D       ← cumulative! errors compound
Line 12935:     pa = panew
            }
```

### Bug 2 (SECONDARY): Hard 180° Proximity Threshold — Lines 12981, 12993

**What it is:**
The check `fabs(p2RT - p2RTLoLim) <= 180.0` is a hard binary decision with no
hysteresis. Its purpose is to reject "ghost" solutions — hour angles where the
trig says the PA matches, but the probe would actually be at the opposite limit.
When `p2RT` is corrupted by Bug 1, this check becomes essentially random.

Even without Bug 1, when `p2RT` happens to land near exactly 180° from a limit,
tiny changes flip the check between TRUE and FALSE on alternating calls. Each
state picks a different candidate hour angle, producing two different countdown
times — the oscillation pattern seen in strip charts.

**The code:**
```c
/* P2 low limit check — line ~12981 */
if ( fabs( p2RT - p2RTLoLim ) <= 180.0 ) {
    if ( !p2valid || (p2halo[j] - curha) < p2ttl ) {
        p2ttl = p2halo[j] - curha;
        p2valid = 1;
    }
}

/* P2 high limit check — line ~12993 */
if ( fabs( p2RT - p2RTHiLim ) <= 180.0 ) {
    if ( !p2valid || (p2hahi[j] - curha) < p2ttl ) {
        p2ttl = p2hahi[j] - curha;
        p2valid = 1;
    }
}
```

**Why it matters:**
No hysteresis, no dead-band, no memory of the previous decision. When Bug 1
pushes `p2RT` far from reality, this check randomly accepts or rejects solutions,
producing the chaotic output. When Bug 1 is mild, this check can still flip on
its own if `p2RT` happens to be near the 180° boundary.

### Amplifying Factors

**Raw encoder noise (no filtering):**
The P2 rotary table position comes from a live encoder reading with no smoothing:
```
$(ag)p2:RT34Pos → AI record → tcsProbeRTPosns() → global pwfs2RTPosn
```
No running average, no deadband, no SMOO field. This noisy starting point feeds
directly into Bug 1's accumulation.

**Parallactic angle instability near zenith:**
`slaPa()` has a mathematical singularity at the zenith — PA changes extremely
rapidly and can flip by large amounts for tiny HA changes. Two of the four 2024
faults (GNFR-73941 at EL=83°, GNFR-74131 at EL=81°) were near zenith, which
makes the PA changes at each simulation step larger and more prone to boundary
crossings.

**Target stream errors (per GNFR-72825):**
The "constant stair step of movement demand" from target stream errors causes
the global variables (`AzDef`, `RmaDef`, pointing demands) to change slightly
on every 50ms calculation cycle, even during stable tracking. This was first
identified by Adam Smith in 2022 as correlating with the timing jumps.

**Fixed CRCS mode (specific trigger):**
The bug only executes when `r_frame == AZEL_TOPO && p2Following` — the
configuration used for MAROON-X, and previously uncommon before MAROON-X was
installed (REL-4008, 2021). In this mode the probe must rotate to compensate
for field rotation, making the PA-based prediction code active.

**No damping in the EPICS signal path:**
```
tcsLimitTimes()          genSub "times"           ao "p2time"           SAD record
   computes p2ttl  →  writes to VALJ  →  OUTJ (PP) pushes to  →  $(sad)timeToP2lim
                       (no filter)         DOL → OUT (PP MS)         (displayed on TSD)
                                           (no filter)
```
There is no point in this chain where oscillation or chaos could be dampened.

## The Fix: Direct Forward Integration

### Approach

Replaced the entire inverse-trig + coarse-simulation + proximity-check machinery
with a simple forward integration of the parallactic angle rate. This is the
"calculus" approach: step forward in small time increments, add up the probe
rotation at each step, and stop when the accumulated rotation reaches a limit.

No ghost solutions, no `slaDrange()` accumulation over coarse steps, no 180°
proximity check. Just: "how long until the area under the rotation-rate curve
fills up the remaining distance?"

### What Changed

The old code (Phase 1 + Phase 2 in the algorithm description above) has been
**completely removed** and replaced with a standalone forward integration loop
for each probe. The azimuth and rotator wrap calculations are untouched.

**New code (for each probe):**
```c
#define P12_HA_STEP  (5.0 * D2R)       /* 5 deg HA ≈ 20 sec sidereal */
#define P12_N_STEPS  ((int)(PI2 / P12_HA_STEP))  /* ~72 steps = 24h */

p2paLast = pa0;    /* current parallactic angle */
p2rot = 0.0;       /* accumulated rotation from current position */
for ( i = 0; i < P12_N_STEPS; i++ ) {
    p2ha = curha + (i + 1) * P12_HA_STEP;
    p2panew = slaPa(p2ha, dec, tlat);
    p2rot += slaDrange(p2panew - p2paLast) * R2D;
    p2paLast = p2panew;
    p2pos = p2RTPosn + p2rot;

    if ( p2pos <= p2RTLoLim ) {    /* hit the low limit */
        p2ttl = (i + 1) * P12_HA_STEP;
        p2valid = 1;
        break;
    }
    if ( p2pos >= p2RTHiLim ) {    /* hit the high limit */
        p2ttl = (i + 1) * P12_HA_STEP;
        p2valid = 1;
        break;
    }
}
```

### Why This Works

1. **5° HA steps** instead of 90° — the PA change per step is tiny (a few degrees),
   so `slaDrange()` never hits a boundary crossing. The accumulation is safe.
2. **Direct limit check** — no inverse trig producing ghost solutions, no proximity
   check to flip. When `p2pos` crosses a limit, that's the answer.
3. **~72 iterations of one `slaPa()` call each** — negligible CPU cost.
4. **Single answer** — no ambiguity between multiple candidates.
5. **Stable** — tiny changes in starting position produce tiny changes in the answer,
   because the integration is smooth and continuous.

### What Was Removed

- `tcsLimPa()` calls for P1/P2 (the inverse trig that produced ghost solutions)
- P1/P2 candidate HA arrays and adjustment loops
- P1/P2 initialization and done-flag setup in the simulation loop
- The cumulative `p2RT += slaDrange(panew - pa) * R2D` (lines 12928-12929)
- The 180° proximity checks for P1/P2 (lines 12944, 12962, 12981, 12993)
- Unused variable declarations (`p1halo`, `p2hahi`, `p1RT`, `p2RT`, etc.)

## How to Verify

1. Set `INPH=1` on the `times` genSub record to enable debug output
2. Watch the console for:
   ```
   tcsLimitTimes: P2 lower limit HA's (mins): X  Y
   tcsLimitTimes: N  p2RT (degs) = Z
   ```
3. **Before the fix:** `p2RT` should show large variations between steps and
   across calls, potentially exceeding physical limits (±270°)
4. **After the fix:** `p2RT` should track smoothly within the probe's physical
   range, and VALJ should be a monotonic countdown
5. Use `time2unwrapCalcMonitor.py` (in `p2unwrapTroubleshooting/`) to capture
   VALJ at 50ms intervals and verify stability
