# REL-4620 — PWFS-2 Spurious Unwrap / Alternating End-of-Wrap Times

## What's Happening (Plain English)

The telescope has a guide probe called PWFS-2 (P2) that sits on a rotary table.
As the telescope tracks a target across the sky, P2 slowly rotates toward its
mechanical limit — like a clock hand approaching 12. The TCS (Telescope Control
System) predicts *when* P2 will hit that limit and displays a countdown on the TSD
(Telescope Status Display).

**The bug:** Instead of showing one smooth countdown, the TSD alternates between
two different times on every update — for example 15 minutes and 22 minutes, back
and forth like a strobe. Eventually the shorter time reaches zero and the system
triggers an unwrap (reverses P2's direction), even though the real limit was still
far away.

This has happened at least 4 times on-sky (GNFR-73941, GNFR-73994, GNFR-74131,
GNFR-74132), interrupting observations.

---

## How the Countdown is Calculated

The function `tcsLimitTimes()` in `tcsPointing.c` does this roughly:

1. **Read the current state** — P2's current rotary table position (`p2RTPosn`)
   from a live encoder, the current hour angle, declination, etc.

2. **Ask: "At what hour angle will P2 reach its limit?"** — It calls `tcsLimPa()`
   which solves a trig equation. This can return **two answers** (h1 and h2) — two
   different future hour angles where the math says the limit could be reached.
   It does this for both the low limit and the high limit, so there can be up to
   **four candidate answers**.

3. **Simulate forward in time** — Step the hour angle forward in 90° increments
   (6 steps). At each step, project where P2 would be:
   ```c
   p2RT += slaDrange(panew - pa) * R2D;
   ```

4. **Check proximity** — When the simulation passes one of the candidate hour
   angles, check if the projected P2 position is actually close to the limit:
   ```c
   if ( fabs( p2RT - p2RTLoLim ) <= 180.0 ) {
       /* Yes — accept this time as the countdown value */
   }
   ```

5. **Pick the shortest valid time** and write it to `VALJ`, which flows through
   to the TSD display with zero filtering.

---

## The Three Most Likely Causes

### Cause 1: The Proximity Check Flips Between Two Solutions

**What it is:**
The check `fabs(p2RT - p2RTLoLim) <= 180.0` is a hard yes/no boundary. When `p2RT`
is near exactly 180° from the limit, tiny changes flip it between TRUE and FALSE
on alternating calls. Each state picks a different candidate hour angle, producing
two different countdown times.

**Evidence:**
- The strip chart (Aug 14 2024) shows a textbook square wave — the value toggles
  between exactly two levels (~15 min and ~22 min) every single cycle, with no
  intermediate values. This is not random noise; it's a binary flip.
- The gap between the two levels is consistent (~7 min apart) which matches two
  different HA solutions from `tcsLimPa()` — one from the lo-limit path, one from
  the hi-limit path.
- The code uses the **projected** position (`p2RT`) which accumulates noise over 6
  forward steps, not the current measured position. This makes the flip zone wider.

**The code:**
```c
/* P2 low limit check — line ~12981 */
if ( fabs( p2RT - p2RTLoLim ) <= 180.0 ) {
    if ( !p2valid || (p2halo[j] - curha) < p2ttl ) {
        p2ttl = p2halo[j] - curha;   /* pick this solution */
        p2valid = 1;
    }
}

/* P2 high limit check — line ~12993 */
if ( fabs( p2RT - p2RTHiLim ) <= 180.0 ) {
    if ( !p2valid || (p2hahi[j] - curha) < p2ttl ) {
        p2ttl = p2hahi[j] - curha;   /* pick this solution */
        p2valid = 1;
    }
}
```

**Why it matters:**
There is no hysteresis, no dead-band, no "sticky" behavior. The threshold is a
razor edge, and the input is noisy enough to straddle it.

---

### Cause 2: Raw Encoder Noise on P2 Position

**What it is:**
The P2 rotary table position (`p2RTPosn`) comes from a live encoder reading
(`$(ag)p2:RT34Pos`) with **no filtering or smoothing** anywhere in the chain.
Even a few degrees of jitter changes where the forward simulation starts, which
shifts `p2RT` at the point where the proximity check fires.

**Evidence:**
- The data flow has zero filtering at every stage:
  ```
  $(ag)p2:RT34Pos → AI record → tcsProbeRTPosns() → global pwfs2RTPosn
  ```
  No running average, no deadband, no SMOO field on the AI record.
- `p2RT` starts at the raw encoder value and then accumulates PA changes:
  ```c
  p2RT = p2RTPosn;                          /* raw encoder */
  p2RT += slaDrange(panew - pa) * R2D;      /* accumulated 6 times */
  ```
  If `p2RTPosn` jitters by ±2°, after 6 accumulation steps the error in `p2RT`
  can be larger — the `slaDrange()` wrapping can amplify small differences.
- The `p2time` ao record (visible in the schematic) also has no SDIS/deadband
  that would suppress small output changes. It's a pure pass-through.

**Why it matters:**
Even if the proximity threshold were perfect, noisy input makes the projected
position unreliable. The simulation is deterministic but the starting point isn't
stable.

---

### Cause 3: Parallactic Angle Instability Near Zenith

**What it is:**
The parallactic angle (PA) — the angle between "up" and "north" as seen from the
telescope — is computed by `slaPa(curha, dec, tlat)`. Near the zenith, the PA has
a mathematical singularity: it changes extremely rapidly and can flip by large
amounts for tiny changes in hour angle. The code uses PA to compute the limit
offsets and to project P2 forward:

```c
pa0 = slaPa(curha, dec, tlat);       /* current PA — unstable near zenith */
dRTLoLim = p2RTLoLim - p2RTPosn;
pa = pa0 + dRTLoLim * D2R;           /* limit PA — inherits the instability */
p2locount = tcsLimPa(cdec, sdec, pa, clat, slat, p2halo);  /* solve for HAs */
```

If `pa0` jumps between calls, the PA fed to `tcsLimPa()` changes, which changes
which HA solutions are returned and how many (0, 1, or 2).

**Evidence:**
- GNFR-74131 was at **EL = 80.8°** — very close to zenith where PA is unstable.
- GNFR-73941 was at **EL = 83.2°** — even closer to zenith.
- GNFR-73994 was at EL = 48° — not near zenith, so this cause alone doesn't
  explain all incidents. But it amplifies Cause 1 when it does apply.
- The forward simulation also calls `slaPa()` at each step:
  ```c
  panew = slaPa(ha, dec, tlat);
  p2RT += slaDrange(panew - pa) * R2D;
  ```
  Near-zenith targets make these intermediate PA values jumpy too, inflating the
  projected `p2RT` error.

**Why it matters:**
For 2 of the 4 faults, the target was near zenith. This doesn't cause the bug by
itself — Cause 1 is the gate — but it makes the flip zone much wider so the bug
triggers more easily.

## How the Causes Work Together

```
                         Cause 2                    Cause 3
                    (encoder noise)          (PA instability near zenith)
                           |                          |
                           v                          v
                     p2RTPosn jitters           pa0 jumps between calls
                           |                          |
                           +--------+--------+--------+
                                    |
                                    v
                          p2RT (projected position)
                          is unstable near 180° boundary
                                    |
                                    v
                              Cause 1
                    (hard threshold, no hysteresis)
                                    |
                        +-----------+-----------+
                        |                       |
                  fabs() <= 180             fabs() > 180
                  pick solution A           pick solution B
                        |                       |
                        v                       v
                   short time              long time
                        \                     /
                         alternating display
                                |
                                v
                    short time hits zero → spurious unwrap
```

Cause 1 is necessary for the bug. Causes 2 and 3 are amplifiers that determine
*when* the probe is close enough to the 180° boundary for the flip to happen.

## EPICS Signal Path (No Damping)

```
tcsLimitTimes()          genSub "times"           ao "p2time"           SAD record
   computes p2ttl  →  writes to VALJ  →  OUTJ (PP) pushes to  →  $(sad)timeToP2lim
                       (no filter)         DOL → OUT (PP MS)         (displayed on TSD)
                                           (no filter)
```

There is no point in this chain where the oscillation could be dampened.
The fix must be at the source.

## Proposed Fix Options

### Option A — Use current position + add hysteresis margin (simplest)

```c
/* BEFORE — uses projected p2RT, hard 180° boundary */
if ( fabs( p2RT - p2RTLoLim ) <= 180.0 ) {

/* AFTER — uses current p2RTPosn, 10° hysteresis band */
if ( fabs( p2RTPosn - p2RTLoLim ) <= 190.0 ) {
```

- `p2RT` → `p2RTPosn`: stop using the accumulated projection, use what the
  encoder says right now
- `180.0` → `190.0`: add a 10° margin so noise can't flip the check

Apply to all 4 probe checks (P1 lo, P1 hi, P2 lo, P2 hi).

### Option B — Low-pass filter on encoder input

Add smoothing in `tcsProbeRTPosns()` before storing to the global. Addresses
the noise at the source but adds state and needs a tuning parameter.

### Option C — Require consistent solutions before updating output

Add `static` counters — only update the output if the same solution wins N
times in a row. Most robust but most complex.

**Recommendation:** Start with Option A. It's minimal, testable, and directly
addresses Cause 1. If encoder noise is still a problem after that, layer in
Option B.

---

## How to Verify

1. Set `INPH=1` on the `times` genSub record to enable debug output
2. Watch the console for:
   ```
   tcsLimitTimes: P2 lower limit HA's (mins): X  Y
   tcsLimitTimes: N  p2RT (degs) = Z
   ```
3. If `p2RT` oscillates near 180° from the limit while the output alternates,
   Cause 1 is confirmed
4. After the fix, the output should be a smooth monotonic countdown
